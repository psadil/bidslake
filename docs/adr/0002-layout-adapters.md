# ADR 0002 — Layout adapters: querying standardized non-BIDS datasets

Status: accepted (2026-07-14)

Relates to: [ADR 0001](0001-schema-augmentation-overlays.md) (supersedes its §6).

## Context

A great deal of real neuroimaging data lives in outputs that are *standardized but not
BIDS*: FreeSurfer `recon-all` (the immediate target), HCP-style directory trees, and the
`sourcedata/freesurfer` tree that fMRIPrep itself ships. Being able to query these next to
BIDS data — in one catalog, by one set of concepts — is arguably bidslake's core advantage.

Overlays (ADR 0001) cannot reach them, and it is worth being precise about *why*, because it
is the whole justification for this ADR. An overlay extends the BIDS **vocabulary**: new
entities, suffixes, columns, table shapes. But every generator downstream still identifies a
file by parsing BIDS entities **out of its filename**. `stats/aseg.stats` and
`surf/lh.thickness` have no entities to parse — their identity is carried by their
**position in the directory tree** (`<subject>/stats/aseg.stats`). No amount of added
vocabulary fixes a file whose name says nothing.

So there were two separable gaps:

1. **Projection** — map a non-BIDS *path* onto BIDS *concepts* (`sub`, `ses`, `datatype`,
   `suffix`). This is the genuinely new capability; overlays cannot express it.
2. **Disposition** — decide what bidslake *does* with a file once projected: parse its
   contents into a data table, catalog it and leave it on disk, or skip it. This is not a
   BIDS question at all (BIDS has no database), and for plain BIDS it was scattered across
   hardcodes (`is_tabular_file`, a literal `.bval`/`.bvec` branch, a `.tsv.gz` special case,
   `is_order_insensitive`).

An earlier prototype solved both with a single bespoke DSL (`x_bidslake` keys, `action:
read`, `x_bidslake_tables`) hung off the schema. It worked, but it had no path into any
standard, required its own parser and validator, and conflated concerns that turn out to be
independent.

## Decisions

### 1. An adapter is three artifacts, not one DSL

`--adapter freesurfer` resolves a **trio** of separately-validatable documents plus reader
code. Each answers exactly one question:

| Layer | Artifact | Answers | Metaschema | Headed upstream? |
|---|---|---|---|---|
| 1. **Overlay** | `data/overlays/freesurfer.json` | *What vocabulary and table shapes exist?* (entities `hemi`/`seg`/`parc`, suffixes `segstats`/`parcstats`, typed columns, `rules.tabular_data.freesurfer.*`) | BIDS metaschema (existing) | **Yes** — an ordinary overlay, per ADR 0001 |
| 2. **Term map** | `data/term-maps/freesurfer.json` | *What BIDS concepts does this path denote?* | new `term-mapping-metaschema.json` | **Yes** — pure BEP-043 |
| 3. **Ingestion schema** | `data/ingestion/freesurfer.json` | *What does bidslake do with the file?* (read / catalog / ignore, which reader, per-table policy) | new `ingestion-metaschema.json` | **No** — deliberately bidslake-specific |

The decomposition is the point. Layer 1 is *already* a solved problem (ADR 0001), so
FreeSurfer's tables are generated by the **existing** overlay path with no new code. Layer 2
is the one new capability. Layer 3 is bidslake's own concern and is therefore not proposed
to anyone — but it is still *formalized* rather than hardcoded (see §4).

*Rejected:* the `x_bidslake` DSL — one artifact, three concerns, no standards path, bespoke
parser. *Rejected:* LinkML for the new schemas — BIDS itself uses hand-written JSON Schema;
LinkML in BIDS is a single unmerged experimental PR.

### 2. Term maps are pure BEP-043, with PCRE templates

Layer 2 is exactly BIDS Extension Proposal 043 ("BIDS Term Mapping") — `BIDSVersion` /
`BIDSMapVersion` / `Mappings[]`, each with a `Template` plus `Entities` / `Concepts` /
`Metadata`. We add **no** bidslake keys, so a bundled term map is a valid BEP-043 document we
could contribute upstream.

BEP-043 floats two `Template` syntaxes (`{var}` interpolation and PCRE) and has not settled.
We pin **PCRE**, because named capture groups collapse the combinatorics: one template

```
(?:sub-)?(?P<subject>[0-9A-Za-z]+)(?:_ses-(?P<session>[0-9A-Za-z]+))?/stats/aseg\.stats
```

matches all three subject-directory conventions in the wild (`sub-01_ses-1`, `sub-01`, bare
`01`/`bert`) — roughly a 3× drop in mappings versus enumerating `{var}` forms. Named groups
bind to entities (`subject`→`sub`, `session`→`ses`); `Concepts` set `datatype`/`suffix`
literally. The `regex` crate covers the needed subset (no lookaround or backreferences).

BEP-043 has no official JSON Schema, so we wrote one (`term-mapping-metaschema.json`).

Implementation: `bids_schema::term_map` — `TermMap::classify(rel_path) -> Option<FileFacts>`,
the analog of `read_entities` for non-BIDS paths. It is IO-free and shared, so the validator
can use it too (§11).

### 3. Term maps are anchored; a nested tree is its own dataset

`classify` anchors each template (`^(?:…)$`) against the **dataset-relative** path. The
anchor is deliberate — an unanchored search would happily read `freesurfer` or `sourcedata`
as a subject label.

The consequence is worth stating plainly, because it surprises people: a FreeSurfer tree
nested at `fmriprep/sourcedata/freesurfer/` is **not** picked up by indexing the fMRIPrep
root. (Not because `sourcedata` is opaque — bidslake's *ingest* walk has no opaque-directory
pruning, unlike the validator, so the files are reached — but because the
`sourcedata/freesurfer/` prefix defeats the anchor.) Index it as **its own dataset**:

```
bidslake index --input <fmriprep>/sourcedata/freesurfer --adapter freesurfer --dataset-id freesurfer --output study.duckdb
```

Multiple datasets accumulate in one database (`CREATE TABLE IF NOT EXISTS`, open-not-truncate),
so both end up queryable side by side, distinguished by `dataset_id`. This is the honest
model anyway: the recon-all tree *is* a separate dataset that fMRIPrep happens to vendor.

### 4. The ingestion schema: bidslake's own concern, formalized (supersedes ADR 0001 §6)

ADR 0001 §6 declined to invent a BIDS `row_order` concept and left the policy **hardcoded**
in `bids::is_order_insensitive`. That reasoning was right about BIDS and wrong about
bidslake: "does this file get read into a table, and does its row order matter" is not a BIDS
question — BIDS has no database — so there was never a need to invent a *BIDS* concept. There
was a need for a **bidslake** schema.

So read-vs-catalog is now its own formal, metaschema-validated layer. It is **not**
adapter-only: `data/ingestion/base.json` is the BIDS default, applied on every ingest:

```jsonc
{ "rules": [
    { "selectors": ["intersects([extension], [\".bval\", \".bvec\"])"], "disposition": "read", "reader": "diffusion" },
    { "selectors": ["extension == \".tsv.gz\""],                        "disposition": "catalog" },
    { "selectors": ["extension == \".tsv\""],                           "disposition": "read", "reader": "csv" } ],
  "tables": { "events": { "ordered": false } } }
```

An adapter merely contributes a **fragment** that is merged onto this base. The rules select
with the **BIDS selector-expression language** over *projected* concepts, reusing the very
same evaluator as `Tabular::route` — so even bidslake's private layer refers back to the BIDS
schema rather than inventing a vocabulary.

This retires `is_order_insensitive`, `is_tabular_file`, the literal `.bval`/`.bvec` branch,
and the `.tsv.gz` special case. `bids-2-devel#98` remains the upstream home for row-order *if*
BIDS ever wants it; bidslake no longer waits on it.

Implementation: `bidslake::schema::ingestion::Ingestion` — `classify(&FileContext) ->
Option<&IngestionRule>`, `ordered(table)`, `materialized_concepts(table)`.

### 5. The schema decides; the existing engines execute

The ingestion schema was the riskiest piece, because the BIDS `.tsv` path is **batched**: files
are grouped by header and ingested with one `read_csv` over many (the "Lever 1b" win). The
adapter readers, by contrast, are per-file. Routing `.tsv` through a per-file reader — the
naive reading of "make it schema-driven" — would have un-batched the hot path.

So the schema drives the **decision** (disposition + reader *name*) while the existing
**engines** perform the work: `reader: "csv"` means *the batched tabular ingest*, `reader:
"diffusion"` means *the bval/bvec accumulator*, `reader: "fs_stats"` means *a per-file
`ContentReader`*. Nothing in the fast path was rebuilt.

Two guards keep it honest: primary data files short-circuit **structurally** (`is_datafile`)
*before* the ingestion dispatch, keeping per-file selector evaluation off the imaging-file hot
path; and `datatype` is deliberately **not** bound at the BIDS dispatch, so a configured
adapter's datatype-keyed rules cannot claim ordinary BIDS files. Net effect measured: ingest
got ~3–8% *faster* (the JSON short-circuit also skips a per-file `find_datatype`), and the
selector-AST cache that this work motivated made validation ~25–30% faster.

### 6. Three dispositions — and the catalog surface follows the file, not the disposition

`read` (parse contents into a data table via the named reader), `catalog` (register it,
contents unread, left on disk), `ignore` (skip — a declarative `.bidsignore` override).

`catalog` means *register but do not read*; **which registry** depends on what the file is,
not on the disposition: a primary data file lands in `scans` (surfaced as the `files` view), a
tabular file lands in `tabular_files`. That is why compressed continuous recordings
(`*_physio.tsv.gz`) are `catalog`, not `ignore` — they are far too large to ingest row-per-sample,
but users must still be able to *find* them in order to read them with e.g. polars.

*Rejected:* an `identity` field (`per_file`/`per_row`/`per_entity`). It conflated three things
that are already structural — `scans` is always per-file, a `read` is always per-row, and
per-entity is a BIDS relational notion expressed by `index_columns`. *Rejected:* `sort_by` —
only `ordered` is needed. *Rejected:* `$schema` fields on the artifacts — the URIs are unhosted,
so they help no IDE, and the BIDS metaschema's top-level `additionalProperties: false` forbids
one on an overlay anyway.

### 7. Materialized concept columns for adapter tables; BIDS keeps virtual ones

BIDS tables get their entity columns as **virtual** generated columns (regexes over
`file_path`) — cheap, and correct because a BIDS filename *contains* its entities. A
FreeSurfer path does not, so an adapter table needs **physical** `TEXT` columns populated from
the projection. The ingestion schema's per-table `concepts` list is what marks a table
materialized (`freesurfer_aparc: {concepts: ["sub","ses","hemi","parc"]}`). Both kinds are
emitted by the one shared `generate_tabular_table`; plain BIDS ingest is untouched and pays
nothing.

### 8. Hand-written JSON Schema metaschemas, validated per artifact

Both new layers get a metaschema (`term-mapping-metaschema.json`, `ingestion-metaschema.json`),
hand-written JSON Schema draft 2020-12 and checked with the `jsonschema` crate — mirroring ADR
0001 §3. Each artifact is validated **on its own**, so a malformed term map fails with a
message about term maps rather than corrupting a merged blob.

### 9. Self-describing databases (as ADR 0001 §4)

Alongside `bidslake_overlays`, a catalog records `bidslake_term_maps` and `bidslake_ingestion`.
All three layers of an adapter travel with the data.

### 10. Every dataset records a `root_uri` — even one with no `dataset_description.json`

`root_uri` is what turns a stored dataset-relative `file_path` back into an openable
`file://`/`s3://` URI, and it lives on `dataset_description`. But an adapter dataset has no
`dataset_description.json` *by definition* — that is what makes it non-BIDS — so it got no row,
no `root_uri`, and bidslake-py could not resolve a single one of its files.

The ingest now synthesizes a minimal `{dataset_id, root_uri}` row when the walk found none. It
is emitted **after** the walk: the insert carries a `WHERE NOT EXISTS` primary-key guard, so a
bare row written up front would shadow a real `dataset_description.json` and silently drop its
metadata. (Across *runs* the table remains first-writer-wins on `dataset_id`; see the `eh-04`
TODO.)

### 11. The validator consults term maps

A file a term map recognizes is *expected*, not an error. `ValidatorConfig.adapters` names the
bundled term maps; an otherwise-unmatched file that a term map claims suppresses the
`NotIncluded` issue. Recognition walks each `/`-suffix of the path, so a nested derivative
(`derivatives/fmriprep/sourcedata/freesurfer/…`) still matches a subject-anchored template —
the one place we deliberately relax §3's anchoring, because the validator is only asking "is
this file expected?", not "what does it denote?".

## How the layers relate

- **Overlay ⟶ vocabulary.** Answers *what can exist*. Shared with plain BIDS derivatives
  (fMRIPrep/MRIQC/QSIPrep use overlays with no adapter at all). Metaschema-conformant BIDS.
- **Term map ⟶ projection.** Answers *what this path denotes*. Only needed when filenames do
  not carry entities — i.e. only for adapters. A community standard (BEP-043).
- **Ingestion schema ⟶ action.** Answers *what bidslake does about it*. Needed for **every**
  dataset, BIDS or not; adapters just add a fragment. Deliberately bidslake-specific, but
  selector-driven and metaschema-validated rather than hardcoded.

An adapter is precisely the case where all three are needed at once. An overlay alone suffices
for a BIDS-named derivative. The ingestion schema alone is what plain BIDS uses.

## Consequences

- FreeSurfer `recon-all` is queryable as typed tables (`freesurfer_aseg`, `freesurfer_aparc`,
  `freesurfer_measures`, `freesurfer_labels`) joined to BIDS data by `sub`/`ses` in one
  catalog, with no bespoke code per dataset — only three declarative documents plus a content
  reader for the one thing that is irreducibly code: parsing a `.stats` body.
- Adding another layout (HCP, …) is authoring a trio, not writing an ingester.
- Two of the three layers are standards-track; the third is honest about not being.
- Content parsing stays code. The ingestion schema formalizes *which* reader and *how*
  (ordering, materialized concepts), but a multi-table reader's internal routing (`fs_stats` →
  aseg vs aparc, by inspecting `ColHeaders`) remains reader-internal.
- Selecting on a *projected* `suffix` rather than a computed extension is load-bearing:
  `filename_extension` uses BIDS first-dot semantics, so `lh.aparc.stats` has extension
  `.aparc.stats`, not `.stats`. Ingestion rules key on the suffix the term map projects.
- PCRE is one of two syntaxes BEP-043 floats. If the BEP settles on `{var}`, our term maps
  need a rewrite (versioned by `BIDSMapVersion`); the engine boundary is small.
